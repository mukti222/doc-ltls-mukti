Alur Sync & Eligibility AP Import

Dokumentasi teknis — bagaimana sistem menentukan PO yang eligible untuk masuk antrian approval.

1. Trigger Sync

Ada dua cara sync dipicu:

Cara Kode
Auto performApSync({ auto: true }) di $(document).ready()
Manual Konfirmasi SweetAlert → performApSync({ auto: false })

Bisa juga sync per PO dengan mengisi input "PO (opsional)" lalu klik Sync.

2. Flow Sync — performApSync() → Backend

Frontend mengirim POST ke:

Halaman Endpoint
/ap_import {{ url('ap-import/sync') }} → ApImportController@sync()
/ap_import_dev {{ url('ap-import-dev/sync') }} → ApImportDevController@sync()

Kedua controller memanggil method yang sama persis: 

// Jika ada PO spesifik
$result = $apImportSyncService->syncPo($purchasingDocument);

// Jika sync all
$result = $this->apImportService->syncAll();  // internalnya panggil syncPo loop

3. Sync All — syncAll()

syncAll()
  |

4. Inti: evaluatePo() — Menentukan Eligibility

Ini adalah jantung logika sync. Method ini menentukan apakah suatu PO eligible (memenuhi syarat) untuk masuk ke antrian approval. 

evaluatePo($purchasingDocument)
|
+-- Cari sap_header berdasarkan purchasing_document
|   +-- Jika tidak ada → X NOT ELIGIBLE (reason: sap_header_not_found)
|
+-- Cari EtblPibHdr (PIB header)
|   +-- Jika tidak ada → X NOT ELIGIBLE (reason: pib_header_not_found)
|
+-- CEK TT-ADV-Z000 ------------------------------------+
|   Jika payment_method == 'TT'                          |
|   DAN payment_type   == 'ADV'                         |
|   DAN payment_term   == 'Z000'                        |
|   → X NOT ELIGIBLE (PMK / Pembayaran di Muka)      |
|     (reason: tt_adv_z000)                             |
+--------------------------------------------------------+
|
+-- CEK DOKUMEN 380 (INVOICE) -----------------------------+
|   Ambil SEMUA dokumen 380 untuk header ini              |
|   Jika ada:                                              |
|     Loop tiap dokumen 380:                               |
|       +-- isLocked == true?                              |
|       |   Tidak → X NOT ELIGIBLE                      |
|       |     (document_380_not_locked)                    |
|       |                                                    |
|       +-- Sudah upload final document?                   |
|       |   (via Vendor Portal API)                        |
|       |   Tidak → X NOT ELIGIBLE                      |
|       |     (document_380_final_attachment_missing)      |
|       |                                                    |
|       +-- Semua ok → lanjut                           |
|   Jika tidak ada dokumen 380 → skip check            |
+----------------------------------------------------------+
|
+-- Dapatkan vendor_code dari SAP header
|
+-- Bangun SNAPSHOT data PO:
|   * Pilih SnapshotBuilder sesuai shipment_mode (AIR atau SEA)
|   * Snapshot = hash SHA-256 dari seluruh payload data PO
|     (item amounts, qty, transport documents, dll)
|   * Snapshot dipakai nanti untuk deteksi perubahan data
|
+-- CEK TRIGGER TYPE (ada 2 jalur):
|
|  +-- [A] INVOICE DATE TRIGGER ------------------------+
|  |  Cek: usesInvoiceDateTrigger($vendorCode)          |
|  |  → ApImportVendorInvDate::where('vendor_code',   |
|  |       $vendorCode)->where('is_active', 1)          |
|  |                                                    |
|  |  Jika TRUE:                                        |
|  |   * Cari dokumen 380 (Invoice)                     |
|  |   * Jika dokumen 380 tidak ada / DokTg             |
|  |     kosong → X NOT ELIGIBLE                     |
|  |     (invoice_date_missing)                         |
|  |   * triggerDate = DokTg dari invoice               |
|  |   * due_date = triggerDate + paymentTermDays       |
|  |   * V ELIGIBLE                                     |
|  +----------------------------------------------------+
|
|  +-- [B] TRANSPORT DOCUMENT TRIGGER -------------------+
|  |  Jalur ini untuk vendor NON invoice-date           |
|  |                                                    |
|  |   * Ambil SEMUA transport document                 |
|  |     (AWB untuk AIR, B/L untuk SEA)                 |
|  |   * Jika tidak ada → X NOT ELIGIBLE              |
|  |     (transport_doc_missing)                        |
|  |                                                    |
|  |   * Loop tiap transport document:                  |
|  |     +-- Jika isLocked == false → X NOT ELIGIBLE    |
|  |     |   (transport_doc_not_locked)                 |
|  |     +-- Cek attachment type:                      |
|  |     |   - paymentTermDays < 30 → butuh "draft"   |
|  |     |   - paymentTermDays >= 30 → butuh "final"  |
|  |     |   Pengecekan ke Vendor Portal API:           |
|  |     |   GET /api/po-document-file/{tblPibdokId}    |
|  |     |   Jika tidak ada → X NOT ELIGIBLE          |
|  |     |   (attachment_{type}_missing)                |
|  |     +-- Jika DokTg kosong → X NOT ELIGIBLE       |
|  |         (bl_date_missing)                          |
|  |                                                    |
|  |   * Semua ok → triggerDate = DokTg doc pertama   |
|  |   * due_date = triggerDate + paymentTermDays       |
|  |   * V ELIGIBLE                                     |
|  +----------------------------------------------------+
|
+-- Return array berisi:
     eligible => true/false
     supplier_name
     vendor_code
     payment_term_raw, payment_term_days
     shipment_mode (AIR/SEA)
     trigger_doc_kd (380/705/740)
     trigger_attachment_type (invoice_date/draft/final)
     bl_date (triggerDate)
     due_date
     snapshot_hash (SHA-256 dari payload)
     snapshot_payload (array data lengkap PO)

5. Setelah Dinyatakan Eligible — syncPo() 

syncPo($po)
|
+-- evaluatePo($po)
|
+-- Jika TIDAK eligible → return { synced:false, eligible:false, reason }
|
+-- Jika ELIGIBLE:
|
|  +-- PO BARU (belum ada di ap_import_queue) ----------+
|  |  * INSERT queue (status: pending_review)           |
|  |  * INSERT history "Pending Review"                 |
|  |  * return { action: 'created', ... }               |
|  +----------------------------------------------------+
|
|  +-- PO SUDAH ADA -------------------------------------+
|  |  * Cek snapshot hash:                              |
|  |    hash_lama vs hash_baru                          |
|  |                                                    |
|  |  +-- HASH BERUBAH ------------------------------+  |
|  |  |  * Update data queue                         |  |
|  |  |  * Reset status ke PENDING                   |  |
|  |  |  * Jika sebelumnya APPROVED/REJECTED         |  |
|  |  |    → clear approved_at/rejected_at          |  |
|  |  |  * Clear & re-insert history                  |  |
|  |  |  * action: 'updated' atau 'reopened'          |  |
|  |  +----------------------------------------------+  |
|  |                                                    |
|  |  +-- HASH SAMA, status PENDING -----------------+  |
|  |  |  * Update data queue (minor)                 |  |
|  |  |  * action: 'updated'                         |  |
|  |  +----------------------------------------------+  |
|  |                                                    |
|  |  +-- HASH SAMA, status FINAL -------------------+  |
|  |  |  (approved / rejected)                       |  |
|  |  |  * Tidak update apa-apa                      |  |
|  |  |  * action: 'unchanged'                       |  |
|  |  +----------------------------------------------+  |
|  +----------------------------------------------------+
|
+-- return array lengkap dengan eligible_items
     (termasuk hash_change proof untuk dev)

7. Flowchart — Alur Lengkap

Ringkasan

Konsep Penjelasan
Eligible PO memenuhi syarat untuk masuk antrian approval — punya dokumen PIB, SAP header, dokumen trigger (invoice 380 / transport AWB/B/L), attachment sesuai payment term, tanggal trigger terisi, dan semua dokumen 380 (invoice) sudah di-isLocked serta sudah upload final document, dan dokumen transport (AWB/B/L) sudah di-isLocked. Kecuali PO dengan TT + ADV + Z000 (PMK) yang otomatis tidak eligible.
Trigger Type Ada 2: Invoice Date (vendor tertentu via tabel ap_import_vendor_inv_date) atau Transport Document (AWB untuk AIR, B/L untuk SEA/LAND).
Due Date triggerDate + paymentTermDays — dihitung dari tanggal dokumen trigger ditambah term pembayaran.
Snapshot Hash SHA-256 dari seluruh data PO (amount, qty, dokumen transport, dll) — dipakai untuk mendeteksi perubahan data antar sync.
Hash Change Proof Hanya di ap_import_dev — menampilkan payload sebelum/sesudah, diff field-by-field, dan status apakah hash berubah.
Action created (PO baru), updated (data berubah, masih pending), reopened (data berubah, PO sudah final jadi direopen), unchanged (tidak ada perubahan).
💡 Catatan: Backend sync logic untuk ap_import dan ap_import_dev adalah identik. Perbedaan hanya di frontend — ap_import_dev menampilkan modal eligible dengan hash proof cards untuk keperluan debugging dan verifikasi integritas data.
